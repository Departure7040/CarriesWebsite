# Estate microsite builder

A single luxury single-property listing page, refactored into a **reusable template
+ deterministic Python builder** so new listing sites can be generated from a JSON
data file.

```
build/estates/new_estate.py          # ONE command: feed -> facts + photos + AI copy -> build
build/estates/estate_template.html   # the page, content replaced by {{TOKENS}} + block markers
build/estates/build_estate.py        # pure-stdlib builder (json + html + re)
build/estates/estate_content.workflow.js  # (alt) generate copy+captions via the Workflow tool
data/estates/<slug>.json             # one file per listing (all page content as data)
site/estates/<slug>/index.html       # generated output (do not hand-edit)
```

---

## One command (recommended)

```
python build/estates/new_estate.py --address "101 Rio Ridge Dr"   # one listing
python build/estates/new_estate.py --all                          # every active listing
```

`new_estate.py` does the whole pipeline end-to-end:

1. **Facts** — pulls the listing from Carrie's Realtor.com agent feed (price, beds,
   baths, sqft, lot, year, garage, type, coords, listing URL) + best-effort MLS #.
2. **Photos** — downloads all photos, de-dupes (md5), renumbers `01..N`.
3. **Compliance** — writes the LREC broker disclosure + Equal Housing blocks
   deterministically (identical and safe every time; no AI).
4. **Copy + captions** — generated by **local Claude Code (`claude -p`)**, which uses
   your logged-in **claude.ai subscription — no API key, no metering**. Grounded strictly
   in the real facts + what the photos actually show (won't invent "waterfront" etc.).
5. **Build** — runs `build_estate.py` and prints the local + tunnel URL.

Requirements: the `claude` CLI on PATH and logged in. Any `ANTHROPIC_API_KEY` in the
environment is intentionally dropped for the AI calls so they bill against the
subscription, not a metered key. The calls run from a neutral temp dir so this project's
hooks/context don't load into them.

Flags: `--no-content` (skip AI → honest facts-only draft), `--no-photos` (reuse the
existing gallery), `--slug <s>`, `--max-photos <n>` (default 60), `--model-copy` (default
`opus`), `--model-caption` (default `haiku`).

The manual path below (hand-author the JSON, then `build_estate.py`) still works and is
what `new_estate.py` automates.

## Estates directory + custom domains

`build/estates/build_estates_index.py` reads every `data/estates/*.json` and writes
`site/estates/index.html` — a card directory of all listings (sorted by price), linked
into the main site nav as **Estates**. Re-run it after adding/removing a listing:

```
python build/estates/build_estates_index.py
```

**One host, many properties.** Every listing is served from the single main site under
`/estates/<slug>/`. There is no per-property hosting.

**Giving a listing its own domain (no new hosting):**
1. Add `"canonical_domain": "101rioridge.com"` to that listing's
   `data/estates/<slug>.json`, and re-run the index builder — its directory card then
   points at the domain.
2. Point the domain at this same host: add the hostname to the Cloudflare tunnel
   (a public-hostname / ingress rule) or a redirect rule that maps
   `101rioridge.com` → `/estates/101-rio-ridge-dr/`. DNS + one route; the page itself
   does not move.

So a purchased domain is just a redirect onto the already-hosted page — buy it, add the
field, add the route. Nothing is rebuilt or re-hosted.

The golden reference the template was derived from lives at
`site/estates/101-rio-ridge-dr/index.html`. Regenerating from the JSON reproduces it
byte-for-byte except that the agent name/title is canonicalized to a single stored
value (Unicode `®`), so the four body spots that used the `&reg;` entity now emit `®`
— visually and semantically identical.

## The 3-step pipeline (adding a new listing)

### 1. Fetch photos
Drop the gallery images into a public directory, zero-padded and in display order:

```
site/assets/img/listings/gallery/<slug>/01.jpg, 02.jpg, … NN.jpg
```

The hero photo and the video poster are chosen from this same set (by filename).
(Optional) a cinematic tour video goes anywhere web-served, e.g.
`site/studio/packages/videos/<slug>-short.mp4`.

### 2. Generate the JSON
Create `data/estates/<slug>.json`. Copy `101-rio-ridge-dr.json` as a starting point
and edit every field (schema below). The `photos` array must list every gallery image
with a descriptive `alt`. Set `media.video` to `null` if there is no tour video — the
builder then omits the entire Video section, its nav link, and its footer link.

### 3. Build
```
python build/estates/build_estate.py <slug>     # one listing
python build/estates/build_estate.py --all       # rebuild every data/estates/*.json
```
Writes `site/estates/<slug>/index.html` (creates the folder if needed) and refuses to
finish if any `{{TOKEN}}` or block marker is left unresolved.

## JSON schema

Top level:

| key            | type   | notes |
|----------------|--------|-------|
| `slug`         | string | URL slug; also the data filename and output folder |
| `wordmark`     | string | nav brand, line 1 |
| `wordmark_sub` | string | nav brand, line 2 (e.g. `Lafayette · Louisiana`) |
| `facts`        | object | see below |
| `agent`        | object | see below |
| `seo`          | object | `title`, `meta_description`, `og_title`, `og_description` |
| `copy`         | object | all prose (see below) |
| `media`        | object | photos / hero / video |
| `compliance`   | object | verbatim legal HTML blocks |

`facts`: `address`, `city`, `state`, `zip`, `price`, `status`, `beds`, `baths`,
`sqft`, `lot` (e.g. `"0.56 acre"`), `year_built`, `garage`, `type_short`,
`property_type`, `mls`, `realtor_url`, `lat`, `lon`.

`agent`: `name_title` (e.g. `"Carrie Billeaud, REALTOR®"`), `brokerage`, `phone`,
`phone_tel` (digits only), `headshot` (path), `bio` (one paragraph).

`copy`: `hero_eyebrow`, `hero_address`, `hero_tagline`, `overview_eyebrow`,
`overview_heading`, `overview_paragraphs[]`, `pull_quote`, `details_eyebrow`,
`details_heading`, `features_eyebrow`, `features_heading`,
`feature_groups[]` (`{title, items[]}`), `gallery_eyebrow`, `gallery_heading`,
`video_eyebrow`, `video_heading`, `location_eyebrow`, `location_heading`,
`location_paragraphs[]`, `location_points[]`, `cta_eyebrow`, `cta_heading`,
`cta_sub`, `footer_brand`, `footer_brand_sub`.

`media`: `hero_photo` (filename), `hero_alt`, `video` (path **or `null`**),
`video_poster` (filename), `gallery_dir` (path, trailing slash),
`photos[]` (`{file, alt}` — all of them).

`compliance`: `broker_disclosure_html` (the exact LREC/eXp footer disclosure block,
including the `<span data-broker>to be confirmed at launch</span>` placeholders) and
`fair_housing_html` (the Equal Housing Opportunity notice). These are injected
**verbatim** — write them as real HTML strings (they are listing-specific: they carry
the address/price/MLS and must be regenerated per listing).

Text fields (headings, prose, alts, feature items) are stored **raw** — the builder
HTML-escapes `&`, `<`, `>` at build time. Do not pre-escape (write `Arrival & Everyday`,
not `Arrival &amp; Everyday`). URLs, paths, phone digits, and the compliance blocks are
inserted without escaping.

## What the builder computes (do NOT put these in the JSON)

Derived deterministically from the fields above:

- **Spec grid** (Price/Status/Bedrooms/Bathrooms/Living Area/Lot Size/Year Built/
  Garage/Type/MLS #) and **hero spec strip** (beds/baths/sqft/lot number).
- **Overview paragraphs**, **feature cards**, **location paragraphs**, **location
  points** — expanded from their arrays.
- **Photos JS array** `var PHOTOS=[{f,a},…]` (alts JSON-escaped) and **`var LAYOUT`**
  using the 7-tile mosaic cycle `["wide","","half","half","","",""]` truncated to the
  photo count.
- **Lightbox counter** default `1 / <N>`.
- **OSM map** iframe: `bbox` = a box `±0.01°` lon / `±0.005°` lat around `lat,lon`,
  plus `marker=lat,lon`.
- **Meta**: `geo.position` (`lat;lon`), `geo.region` (`US-<state>`), `geo.placename`
  (from `wordmark_sub`), author (`name_title — brokerage`).
- **Derived strings**: agent role line, headshot alt, footer contact line, form
  `listing_address`, message-textarea default, map title, confirm-message name, and the
  `Tagged:` line — all built from `facts`/`agent`.

Output uses CRLF line endings to match the reference page.

## Template internals (for maintainers)

- Scalars are `{{TOKEN}}` placeholders.
- Generated multi-line regions are unique HTML/JS comment markers the builder replaces:
  `<!--SPEC_STRIP-->`, `<!--SPEC_GRID-->`, `<!--OVERVIEW_PARAGRAPHS-->`,
  `<!--FEATURE_CARDS-->`, `<!--LOCATION_PARAGRAPHS-->`, `<!--LOCATION_POINTS-->`,
  `<!--FAIR_HOUSING-->`, `<!--BROKER_DISCLOSURE-->`, `/*PHOTOS*/`, `/*LAYOUT*/`.
- The Video section, its nav link, and its footer link are each wrapped in
  `<!--{{#VIDEO}}--> … <!--{{/VIDEO}}-->`. With a video the markers are stripped; with
  `media.video: null` the whole region is removed.
- All CSS, JS behavior, and structural fixes from the reference are preserved verbatim:
  the lightbox lives **outside** `</main>`; the reveal observer uses `threshold:0` with
  the `setTimeout(…,1200)` safety-net; the gallery grid is `class="grid"` (no `reveal`);
  the favicon, brass-contrast tokens (`#7c5e2b` etc.), `noindex` robots tags, and the
  `data-broker` disclosure placeholders all remain.
